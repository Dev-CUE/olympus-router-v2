# Olympus 세션 프로토콜 (Session Protocol)

> **성격**: 설계 세션의 "일하는 방법" 단일 진실 공급원(SSOT). 설계 내용(PRD/원장)과 분리된 프로세스 규칙.
> **변경 빈도**: 거의 없음(프로토콜). 진행 상태는 HANDOFF.md, 설계 상세는 Ledger, 최종 설계는 PRD가 담당.
> **이 파일을 고치면** 프로세스가 바뀐 것이다. 설계 결정은 여기에 쓰지 않는다.
>
> 📍 **진입점은 `HANDOFF.md` 하나다.** 새 세션은 HANDOFF.md를 먼저 읽고(현재 위치·역할·다음), 그다음 이 문서(일하는 방법)를 읽는다. 이 문서는 "최우선 진입점"이 아니라 "방법론 SSOT"다(06-11 일원화).

## 1. 킵(KIP) & 고잉 프로토콜 (필수 준수)

> 이 프로토콜 위반은 원칙 위반이다. 브리핑 없이 진행하지 않는다.

### 단계

```
1. 브리핑    현재 L-번호 항목의 설계안을 출력한다.
             코드·문서·원장 수정 없음. 출력만.

2. 승인      CUE가 "킵" 또는 "킵하고 다음"이라고 답한다.
             승인 없이 다음 단계로 가지 않는다.

3. 원장 갱신 GitHub MCP로 Olympus_Design_Ledger.md를 직접 수정·푸시한다.
             (Dev-CUE 계정 쓰기 권한 보유 — 직접 푸시 가능)
             갱신 내용: 확정 결정 추가 / 결정 대기 추가 / 충돌 점검 / PRD 반영 메모
             ★ 동시에 HANDOFF.md "현재 위치"(5절)도 갱신한다 (원장만 고치면 핸드오프가 낡는다)

4. 다음 항목 원장·핸드오프 푸시 완료 확인 후 다음 L-번호 브리핑으로 넘어간다.
```

### 절대 금지

- CUE 승인 없이 다음 항목 진행
- 브리핑 없이 설계 확정 선언
- 원장 갱신 없이 "킵 완료" 선언
- 승인 없이 원장·PRD·코드 수정
- 작업 전 브리핑 누락(직접 작업 착수)

### ⛔ 푸시 트리거 규칙 (무단 푸시 방지 — 실제 위반 사례 있음)

> **지적·질문·논의·"~해야 할 것 같다"·누락 보고는 푸시 트리거가 아니다.**
> 이것들은 전부 **브리핑/논의 단계**다. 푸시는 오직 명시적 "킵" / "킵하고 다음" 승인 **직후에만** 한다.

- CUE가 "X가 빠졌다", "X 해야 하지 않나", "X는 어떤가"라고 말하면 → **브리핑/답변**으로 응한다. 푸시하지 않는다.
- "이건 네가 푸시할 일"이라는 **사실 서술**도 푸시 명령이 아니다. 그래도 "킵" 승인을 기다린다.
- "승인"·"좋다"·"오케이" 등도 명시적 "킵"으로 간주하기 애매하면 한 번 확인한다(킵 = 전용 승인어).
- 권한 보유("푸시할 수 있다") ≠ 푸시 허가("푸시해도 된다"). 권한은 항상 있으나, 행사는 승인 후에만.
- 한 번의 "킵하고 다음"은 **현재 브리핑된 항목 1건**의 승인이다. 후속 항목·연쇄 작업을 자동 포함하지 않는다. 묶음 승인이 필요하면 CUE가 명시한다.
- **위반 시**: 즉시 중단, 무단 푸시 사실을 원장 충돌 로그에 기록, CUE에 사후 추인 요청.

### GitHub 갱신 방법

```javascript
// 1) 파일 SHA 먼저 조회
github:get_file_contents({ owner: "Dev-CUE", repo: "olympus-router-v2",
  path: "<파일>", branch: "master" })

// 2) SHA 받아서 업데이트 (전체 파일 내용 재작성)
github:create_or_update_file({ owner: "Dev-CUE", repo: "olympus-router-v2",
  path: "<파일>", branch: "master",
  sha: "<조회한 SHA>",
  message: "docs: confirm L번호 — 항목명",
  content: "<전체 파일 내용>" })
```

> SHA는 매 수정 후 바뀐다. 연속 수정 시 직전 응답의 새 SHA를 사용하거나 재조회한다.
> **SHA 충돌(푸시 실패) 시**: 다른 창/세션이 먼저 갱신한 것이다. 재조회해서 최신 내용을 확인하고, 덮어쓰기 전에 내 변경과 충돌하는지 점검한다. 무작정 재푸시 금지.
> 매 푸시(원장·핸드오프 등) 직전 해당 파일 SHA를 재조회한다(연속 푸시 시 직전 응답 SHA 사용 가능).

## 2. 핸드오프 문서 생성·갱신 규칙

> 핸드오프가 낡거나 부실하면 다음 세션이 망친다. 읽기 게이트만으로는 부족하다.
> **대상 파일 = HANDOFF.md** (진입점 일원화 후 단일).

### 갱신 시점 (언제)

- 매 항목 "킵" 승인 직후, 원장 푸시와 **동시에** HANDOFF.md "현재 위치"(5절)를 갱신한다.
- 세션 종료 전 반드시 HANDOFF.md "현재 위치"가 최신인지 확인한다.

### 필수 필드 (무엇을)

HANDOFF.md는 항상 아래를 최신으로 유지한다:
1. **현재 위치** (5절): 마지막 확정 항목 / 다음 할 항목 / 진행 중 미확정
2. **확정 요약표**: L# / 항목 / 핵심 결정 한 줄
3. **잔여 항목**: L-prefix 순서
4. **문서 지도**: 어느 문서가 무엇을 담나 (진입점 혼선 방지)
5. **SSOT 우선순위**: PRD > 원장 > 핸드오프

### 작성 원칙 (어떻게)

- 핸드오프는 "사람 가독성"이 아니라 "다음 LLM이 오해 없이 파싱"을 기준으로 쓴다.
- 헷갈렸던 구분은 명문으로 박는다(아래 혼동 사전 참조·확장).
- 추측 금지. 확정 안 된 건 "미확정"으로 명시.
- 분량은 한 세션 컨텍스트에 다 들어가게 압축 유지. 상세 근거는 원장으로 민다.

## 3. 혼동 방지 사전 (반복 실수 차단)

> 과거 세션이 실제로 헷갈린 항목. 새 세션은 여기부터 확인한다.

- **진입점은 HANDOFF.md 하나** — 과거 핸드오프 3개(HANDOFF / Design_Handoff_New_Session / Handoff_v610) 중 **HANDOFF.md만 현행**. 나머지는 통합·아카이브(06-11). 다른 핸드오프 파일을 진입점으로 읽지 마라.
- **"수정 금지" 문구의 대상** — CLAUDE.md·AGENT.md·SKILLS.md의 "문서 수정 금지"·"블랙리스트" 조항은 **C(구현 CLI)를 향한 가드**다. A(설계)·B(문서 정합)에게 거는 제약이 아니다 — A/B는 설계·문서 작성이 본업. 자기제약으로 오인 금지.
- **L21 ≠ P21** — `L숫자`=설계 항목(작업 단위), `P숫자`=결정 대기(보류 결정). 다른 체계.
- **테스트 ID 체계** — `T5`=A2A / `T7`=wiki·Obsidian / `T9`=admin / `T10`=큐·전송·재시작 / `TA`=audit / `TR`=rate limit·quota(L21) / `TO`=관측성·어댑터 metrics(L20) / `TS`=저장소·tenant 키·수평전환(L17/18/22) / `TD`=보존·삭제·롤오버·배치(L23). 관심사별로 대역이 분리돼 있다. 신규 테스트는 해당 관심사 대역에 붙인다(아무 대역에나 연장 금지).
- **audit ≠ 시스템 로그** — audit=메시지 내용(컴플라이언스, 토글 가능). 시스템 로그=job 상태 전이(운영, 상시). **독립 레이어, 상호 토글 무관.** 같은 egress라도 audit는 "게시한 텍스트", 시스템 로그는 "전달 상태·재시도·dedup"을 본다.
- **fail-open(유량) ≠ fail-closed(인증)** — 인증(#2)은 보안 게이트라 저장소 장애 시 전 요청 거부(closed). rate limit·quota(L21)는 가용성 보호라 카운터 장애 시 통과(open). 표면상 상충처럼 보이나 **레이어가 다르다**(보안 vs 가용성).
- **A2A speaker_counts ≠ L21 rate limit** — speaker_counts(10회, #6)는 "회의 독주 방지"(A2A 의미 레이어). L21 rate limit은 "인프라 유량 보호"(전송 레이어). 둘 다 한도지만 목적·범위가 별개. 동시 적용된다.
- **persona tenant 격리 ≠ 플랫폼 초월** — persona는 플랫폼은 초월(공유)하되 tenant는 격리(`{tenant}:{agent}`). 축이 다르다(L18). 플랫폼 prefix 금지 원칙은 불변.
- **종료 사유 우선순위 ≠ 가드 검증 순서** — A2A 종료 사유는 resolved 최우선(원장 #6.2 성격). 가드 검증은 보안(자기호출·권한·스푸핑) 먼저 통과 후 resolved 평가(원장 #5 가드 순서). resolved가 "최우선"인 건 종료 사유 축이지 가드 실행 축이 아니다(LB-13).
- **재시작 "핸드셰이크" 없음** — 어댑터↔라우터 재시작 복구는 명시적 악수가 아니라 **양방향 재시도+멱등**으로 처리. 어댑터는 무상태라 재동기화 불요.
- **"킵" = CUE 전용 승인어** — 에이전트가 자칭 "킵 완료" 선언 금지. 승인은 CUE만. 지적·질문은 승인이 아니다(1절 푸시 트리거 규칙).
- **at-least-once 한계** — egress는 exactly-once 불가. 게시→dedup커밋→ACK 순서로 최소화하나, 1회 중복 게시는 구조적으로 0이 아니다(명문 인정).
- **일→월 롤오버는 정책 공통·구현 분리** — 시스템 로그(텍스트 합본)와 audit(DB 세그먼트 audit-YYYYMM.db)는 "일→월"이라는 정책만 같고 구현은 다르다(L23). audit 세그먼트는 prev_hash 경계 인계로 해시 체인 보존.

## 4. 세션 자가점검 체크리스트

### 시작 시
```
□ HANDOFF.md "현재 위치"(5절) 확인했나 (단일 진입점)
□ 이 Session_Protocol.md 읽었나 (일하는 방법)
□ 원장(Ledger)·PRD 읽었나(필요 절)
□ 킵 프로토콜·혼동 사전·푸시 트리거 규칙 숙지했나
□ 내 역할(A/B/C) 판별했나 — 이 채팅은 보통 A 또는 B
```

### 종료 시
```
□ 확정 항목 원장(Ledger) 푸시 완료
□ HANDOFF.md "현재 위치" 갱신 완료
□ 미확정·진행중 항목 명시 완료
```
